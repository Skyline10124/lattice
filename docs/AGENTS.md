## 2. 双文档问题追踪系统

### 2.1 活跃问题文档

**文件**: `docs/review/active-issues.md`
**用途**: 系统记录所有未解决问题

每个问题条目必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| **ID** | ✅ | 格式：`P{等级}-{序号}`，如 `P1-3` |
| **首次报告** | ✅ | 首次发现的日期 |
| **来源** | ✅ | 链接到发现该问题的审查报告 |
| **组件** | ✅ | 所属 crate 或模块 |
| **文件** | 条件 | 相关源码文件路径（如有） |
| **状态** | ✅ | `Open` / `In Progress` / `Blocked` |

问题正文必须包含：

1. **描述**：清晰说明问题是什么
2. **复现步骤**：如何触发该问题（P0/P1 必填）
3. **根因**：5-Why 分析（P0/P1 必填）
4. **影响范围**：受影响的用户/场景
5. **建议修复**：具体的修复方案

### 2.2 已解决问题归档

**文件**: `docs/review/resolved-issues.md`
**用途**: 维护所有已解决问题的完整记录

每个条目必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| **ID** | ✅ | 保持原始 ID 不变 |
| **等级** | ✅ | 原始严重度等级 |
| **组件** | ✅ | 所属 crate 或模块 |
| **解决日期** | ✅ | 修复合并的日期 |
| **Commit** | ✅ | 修复的 commit hash |

问题正文必须包含：

1. **描述**：原始问题描述
2. **根因**：为什么会出现这个问题
3. **解决方案**：具体做了什么修复
4. **验证**：如何确认修复有效

## 3. 格式标准

### 3.1 问题条目模板

活跃问题：

```markdown
### P{等级}-{序号}：{简短标题}

| 字段 | 值 |
|------|-----|
| **ID** | P{等级}-{序号} |
| **首次报告** | YYYY-MM-DD |
| **来源** | [报告文件名](报告文件名.md) |
| **组件** | {crate 名称} |
| **文件** | `{源码路径}` |
| **状态** | Open / In Progress / Blocked |

**描述**：{详细描述}

**复现步骤**：
1. {步骤 1}
2. {步骤 2}
3. {预期结果 vs 实际结果}

**根因**：{5-Why 分析}

**建议修复**：{具体方案}
```

已解决问题：

```markdown
### {ID}：{简短标题}

| 字段 | 值 |
|------|-----|
| **ID** | {原始 ID} |
| **等级** | {原始等级} |
| **组件** | {crate 名称} |
| **解决日期** | YYYY-MM-DD |
| **Commit** | `{hash}` |

**描述**：{原始问题描述}

**根因**：{为什么会出现这个问题}

**解决方案**：{具体做了什么}

**验证**：{如何确认修复有效}
```

### 3.2 严重度等级

| 等级 | 含义 | 响应要求 | 示例 |
|------|------|---------|------|
| P0 | 对外可用性阻断 | 立即修复 | 核心功能不可用、安全漏洞 |
| P1 | 运行时正确性缺陷 | 本迭代修复 | 数据丢失、错误结果、panic |
| P2 | 设计/安全/维护问题 | 计划中 | API 设计缺陷、安全隐患 |
| P3 | 代码质量改进 | 延后 | 缺少测试、命名不佳 |

### 3.3 状态流转

```
Open → In Progress → (验证通过) → 迁移到 resolved-issues.md
                    → (验证失败) → 回到 Open
       → Blocked → (解除阻塞) → In Progress
```

## 4. 版本控制与更新流程

### 4.1 新增问题

1. 在 `active-issues.md` 中按严重度分组添加条目
2. 分配唯一 ID（格式 `P{等级}-{下一个序号}`）
3. 填写所有必填字段
4. 更新文档顶部的"最后更新"日期
5. 更新底部统计表

### 4.2 更新问题状态

1. 修改 `active-issues.md` 中对应条目的 `**状态**` 字段
2. 如有新进展，在条目下方添加 `**进展**` 子节
3. 更新"最后更新"日期

### 4.3 关闭问题

1. 从 `active-issues.md` 中删除该条目
2. 在 `resolved-issues.md` 中添加完整归档条目
3. 更新两个文档的统计表
4. 在归档条目中添加 `**关联活跃 ID**` 字段，记录原始 ID（便于追溯）

### 4.4 审查报告

审查报告按日期命名：`code-review-YYYY-MM-DD.md`

每次审查报告应：
1. 引用 `active-issues.md` 中的当前问题清单
2. 报告新发现的问题（同步添加到 `active-issues.md`）
3. 确认已修复的问题（同步迁移到 `resolved-issues.md`）
4. 在报告的"历史审查对照"表中链接前后报告

## 5. 交叉引用方法

### 5.1 问题 ↔ 审查报告

- 活跃问题的 `**来源**` 字段链接到发现该问题的审查报告
- 审查报告的"历史审查对照"表链接到历史报告文件

### 5.2 活跃 ↔ 已解决

- 问题从 `active-issues.md` 迁移到 `resolved-issues.md` 时，保持原始 ID 不变
- `resolved-issues.md` 中条目保留 `**ID**` 字段，便于搜索追溯
- 如果活跃问题部分修复后仍有残留，在 `active-issues.md` 中新建子 ID（如 `P1-3` 修复后残留 `P3-x`）

### 5.3 问题 ↔ 源码

- 活跃问题的 `**文件**` 字段指向具体源码路径
- 源码中的 `// TODO: P1-3` 注释引用问题 ID
- 修复 commit message 中包含问题 ID（如 `fix(P1-3): UTF-8 safe truncation in Agent::run()`）

### 5.4 问题 ↔ Debug 指南

- `docs/debug/guide.md` 中的"已知 Bug 追踪"表链接到 `active-issues.md` 中的对应条目
- Debug 指南中的排查步骤引用问题 ID

## 6. 存储与检索

### 6.1 文件位置

| 文档 | 路径 |
|------|------|
| 本指南 | `docs/AGENTS.md` |
| 活跃问题 | `docs/review/active-issues.md` |
| 已解决归档 | `docs/review/resolved-issues.md` |
| 调试指南 | `docs/debug/guide.md` |
| 审查报告 | `docs/review/code-review-YYYY-MM-DD.md` |
| 设计构想 | `docs/design/ideas.md` |

### 6.2 检索方法

- **按 ID 搜索**：`grep -rn "P1-3" docs/`
- **按组件搜索**：`grep -rn "lattice-agent" docs/review/active-issues.md`
- **按严重度筛选**：查看 `active-issues.md` 的分组标题
- **按日期追溯**：查看 `resolved-issues.md` 的批次分组

### 6.3 维护责任

| 角色 | 职责 |
|------|------|
| 代码审查者 | 新增问题到 `active-issues.md`，创建审查报告 |
| 修复者 | 更新问题状态为 In Progress，修复后提请验证 |
| 验证者 | 确认修复有效，执行问题关闭流程 |
| 文档维护者 | 定期检查交叉引用完整性，清理过时信息 |
